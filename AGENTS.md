# プロジェクトの目的

cron式を人間向けに変換するMVPを作る

# 開発方針

- MVPは小さく保つ
- 最適化より先に完成させる
- 過剰設計を避ける
- 不明点がある場合は推測で実装せず確認する
- 大きな変更前には実装方針を共有する

# ドキュメント

- 設計ドキュメントは `/docs` 配下に作成する
- Markdown形式で作成する
- ファイル名は機能名にする
- ドキュメントは小さく実用的に保つ

# Rails

- controllerは薄く保つ
- ビジネスロジックはservice objectに切り出す
- Railsの慣習を優先する

# フロントエンド

- ERBのみ使用する
- JavaScriptは最小限にする
- UIはシンプルにする

# 開発フロー

- 30分で作業を止める
- 小さな単位でcommitする

# 品質

- 実装した機能には必ずテストを追加または更新する
- タスク完了前にテストを実行する
- タスク完了前にRubocopを実行する
- Rails Ruby Style Guideに従う
- まずはRails標準機能を優先して使う
- 不要なgemは追加しない
- 可読性と保守性を優先する
